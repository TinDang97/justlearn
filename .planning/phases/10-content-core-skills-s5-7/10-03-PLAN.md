**lesson-01-sql-and-databases-overview.md** — "SQL and Databases: Why Data Engineers Use Them"
Course header: SQL & Databases | 2 hours | Intermediate

Learning Objectives:
- Understand when to use a database vs a CSV file
- Identify the role of SQL in data engineering pipelines
- Recognize SQLite as an embedded database for pipelines and prototyping
- Map database concepts to pandas concepts (table → DataFrame, row → row, column → column)

Part 1: Databases vs Files (30 min)
- Explain when files (CSV/Parquet) are sufficient: small-to-medium data, batch reads, simple pipelines
- Explain when databases are needed: concurrent writes, complex queries, referential integrity, large data that doesn't fit in memory
- Types of databases: relational (SQLite, PostgreSQL, MySQL) vs column-store (BigQuery, Snowflake) vs NoSQL (MongoDB, Redis) — DE uses all of these
- Runnable example (no DB yet): show the problem — reading a 1M row CSV to filter 10 rows is wasteful; SQL can filter at the database level

Part 2: SQL for Data Engineers (30 min)
- Explain: SQL is the query language for relational databases, used to read, write, and transform data
- Core SQL statement types: SELECT (read), INSERT (write), UPDATE (change), DELETE (remove), CREATE/DROP (DDL)
- DE context: most pipeline work uses SELECT (extract), INSERT/UPDATE (load), CREATE TABLE (schema management)
- Explain: SQL databases store data in normalized tables with relationships via foreign keys
- Show entity-relationship example: orders → customers, orders → products (one-to-many)

Part 3: Pandas ↔ SQL Mental Model (30 min)
- Show side-by-side: SQL SELECT vs pandas filtering
  - `SELECT * FROM sales WHERE region = 'North'` ↔ `df[df["region"] == "North"]`
  - `SELECT region, SUM(revenue) FROM sales GROUP BY region` ↔ `df.groupby("region")["revenue"].sum()`
  - `SELECT * FROM orders JOIN customers ON orders.customer_id = customers.id` ↔ `pd.merge(orders, customers, on="customer_id")`
- Explain: SQL runs in the database (server-side), pandas in Python (client-side) — each has strengths
- Show when to use SQL vs pandas: SQL for large data extraction and server-side aggregation, pandas for complex Python logic

Part 4: Hands-on Practice (30 min)
- Exercise 1: Translate 5 SQL SELECT statements to pandas (given SQL, write equivalent pandas code)
- Exercise 2: Translate 5 pandas operations to SQL (given pandas chain, write equivalent SQL)

Key Takeaways: database for concurrent/large data, SQL for server-side query, pandas maps 1-to-1 with SQL concepts, DE pipelines often use both (SQL to extract, pandas to transform)

**lesson-02-sqlite-basics.md** — "SQLite: An Embedded Database for Python"
Course header: SQL & Databases | 2 hours | Intermediate

Learning Objectives:
- Connect to an in-memory SQLite database using sqlite3
- Create tables with CREATE TABLE and appropriate data types
- Insert rows with INSERT INTO
- Query data with SELECT and verify results

Part 1: Connecting and Creating Tables (30 min)
- Show: `import sqlite3; conn = sqlite3.connect(":memory:")`
- Explain :memory: vs file path: :memory: is in-process, lost when connection closes — perfect for learning and testing
- Show CREATE TABLE:
  ```python
  conn.execute("""
      CREATE TABLE employees (
          id INTEGER PRIMARY KEY,
          name TEXT NOT NULL,
          department TEXT,
          salary REAL,
          hire_date TEXT
      )
  """)
  conn.commit()
  ```
- Show SQLite data types: INTEGER, TEXT, REAL, BLOB, NULL (SQLite is loosely typed — discuss briefly)
- Runnable example: create an employees table and a departments table with a foreign key

Part 2: Inserting Data (30 min)
- Show single row insert: `conn.execute("INSERT INTO employees VALUES (1, 'Alice', 'Eng', 90000, '2022-01-15')")`
- Show parameterized insert (always use this — prevents SQL injection): `conn.execute("INSERT INTO employees VALUES (?, ?, ?, ?, ?)", (2, "Bob", "Mkt", 65000, "2023-03-01"))`
- Show executemany for bulk insert:
  ```python
  rows = [(3, "Carol", "Eng", 85000, "2021-06-01"), (4, "Dave", "HR", 55000, "2020-11-15")]
  conn.executemany("INSERT INTO employees VALUES (?, ?, ?, ?, ?)", rows)
  conn.commit()
  ```
- Runnable example: insert 10 employees using executemany with a list of tuples

Part 3: Querying with cursor (30 min)
- Show SELECT with cursor:
  ```python
  cursor = conn.execute("SELECT * FROM employees WHERE salary > 70000")
  for row in cursor:
      print(row)
  ```
- Show cursor.fetchall(): `rows = cursor.fetchall()` — returns list of tuples
- Show cursor.description: `[col[0] for col in cursor.description]` — column names
- Show building a dict from cursor: `[dict(zip(col_names, row)) for row in rows]`
- Runnable example: query all Engineering employees, print name and salary

Part 4: Hands-on Practice (30 min)
- Exercise 1: Create an in-memory database with a products table (id, name, category, price, stock_qty). Insert 8 products using executemany. Query all products where price > 20 and stock_qty > 0.
- Exercise 2: Create orders and customers tables. Insert 5 customers and 10 orders. Query: how many orders does each customer have? (Use GROUP BY in SQL.)

Key Takeaways: sqlite3.connect(":memory:"), CREATE TABLE, INSERT with ? params (never string format), cursor.fetchall(), conn.commit() after writes

Key Takeaways: SELECT cols FROM table WHERE cond ORDER BY col LIMIT n, IN/BETWEEN/LIKE for ranges/patterns, CASE WHEN for conditional columns

**lesson-04-sql-aggregations.md** — "GROUP BY and Aggregations in SQL"
Course header: SQL & Databases | 2 hours | Intermediate

Learning Objectives:
- Use aggregate functions: COUNT, SUM, AVG, MIN, MAX
- Group results with GROUP BY and filter groups with HAVING
- Understand the difference between WHERE and HAVING
- Use subqueries for multi-step aggregation

Part 1: Aggregate Functions (30 min)
- Show COUNT(*), COUNT(col) (excludes NULLs), COUNT(DISTINCT col)
- Show SUM(col), AVG(col), MIN(col), MAX(col)
- Show using aggregates without GROUP BY: `SELECT COUNT(*), AVG(salary) FROM employees`
- Show aggregate on a filtered subset: aggregate + WHERE (filter before grouping)
- Runnable example: total employees, average salary, highest salary, headcount by department

Part 2: GROUP BY (30 min)
- Show GROUP BY with aggregate: `SELECT dept, COUNT(*) as headcount, AVG(salary) as avg_salary FROM employees GROUP BY dept`
- Show GROUP BY multiple columns: `SELECT dept, level, COUNT(*) FROM employees GROUP BY dept, level`
- Show ordering grouped results: `GROUP BY dept ORDER BY avg_salary DESC`
- Show NULL handling in GROUP BY (NULL forms its own group)
- Runnable example: revenue by region and product category (two-level group by)

Part 3: HAVING (30 min)
- Explain HAVING vs WHERE: WHERE filters rows before aggregation, HAVING filters groups after
- Show: `SELECT dept, COUNT(*) as n FROM employees GROUP BY dept HAVING n > 5`
- Show combining WHERE and HAVING: `WHERE hire_date > '2022-01-01' GROUP BY dept HAVING COUNT(*) > 2`
- Show subquery pattern: `SELECT * FROM (SELECT dept, AVG(salary) as avg FROM employees GROUP BY dept) WHERE avg > 75000`
- Runnable example: find product categories with total revenue > 10000 and at least 5 distinct products

Part 4: Hands-on Practice (30 min)
- Exercise 1: Employees — write queries: avg salary by dept, dept with highest headcount, departments where avg salary > 75000, employees earning above their department average (subquery)
- Exercise 2: Sales table (region, product, amount, date) — monthly revenue by region, top 3 products by total sales, regions where total sales > 50000 in the last quarter

Key Takeaways: COUNT/SUM/AVG/MIN/MAX, GROUP BY for aggregation, HAVING to filter groups, WHERE runs before GROUP BY — HAVING runs after

**lesson-05-sql-joins.md** — "SQL JOINs: Combining Tables"
Course header: SQL & Databases | 2 hours | Intermediate

Learning Objectives:
- Write INNER JOIN, LEFT JOIN, RIGHT JOIN (via INNER + swap), and FULL OUTER JOIN
- Join on single and multiple conditions
- Understand table aliases for readable multi-table queries
- Use LEFT JOIN to find non-matching rows (anti-join pattern)

Part 1: INNER JOIN (30 min)
- Explain INNER JOIN: only rows where the join key exists in both tables
- Show syntax: `SELECT o.id, c.name, o.amount FROM orders o INNER JOIN customers c ON o.customer_id = c.id`
- Show table aliases (o, c) for readability
- Show column name conflicts and how to resolve with table.column notation
- Runnable example: join orders + customers, show order details with customer name and email

Part 2: LEFT JOIN (30 min)
- Explain LEFT JOIN: all rows from left table, matched rows from right (NULL if no match)
- Show: `SELECT p.name, COUNT(o.id) as order_count FROM products p LEFT JOIN orders o ON p.id = o.product_id GROUP BY p.id`
- Explain: this finds products with zero orders (order_count = 0) — impossible with INNER JOIN
- Show anti-join: `SELECT p.name FROM products p LEFT JOIN orders o ON p.id = o.product_id WHERE o.id IS NULL`
- Runnable example: list all products and their total order count (include products with 0 orders)

Part 3: Multi-Table Joins and Conditions (30 min)
- Show joining 3+ tables: `orders JOIN customers ON ... JOIN products ON ... JOIN regions ON ...`
- Show non-equi join: `JOIN price_tiers ON employees.salary BETWEEN price_tiers.low AND price_tiers.high`
- Show self-join: `SELECT e.name, m.name as manager FROM employees e JOIN employees m ON e.manager_id = m.id`
- Show JOIN with aggregation: join + GROUP BY pattern to produce summary reports
- Runnable example: enrich a transactions table with 3 dimension tables in a single query, return revenue per customer per region

Part 4: Hands-on Practice (30 min)
- Exercise 1: Write the anti-join: find all customers who have never placed an order. Then find all products that appear in at least 3 different orders.
- Exercise 2: Self-join: employees table with manager_id column. List each employee alongside their manager's name and department. Show employees with no manager (top-level) as well.

Key Takeaways: INNER JOIN = only matches, LEFT JOIN = all left + nulls for non-matches, alias tables for readability, LEFT JOIN + IS NULL = anti-join

**lesson-06-pandas-read-sql.md** — "Reading SQL Query Results into DataFrames"
Course header: SQL & Databases | 2 hours | Intermediate

Learning Objectives:
- Use pd.read_sql() to load query results directly into a DataFrame
- Use pd.read_sql_query() and pd.read_sql_table() appropriately
- Pass query parameters safely to avoid SQL injection
- Read large result sets in chunks using chunksize

Part 1: pd.read_sql() Basics (30 min)
- Show: `df = pd.read_sql("SELECT * FROM employees", conn)`
- Show column names come from the query's column aliases
- Show: `df = pd.read_sql("SELECT dept, AVG(salary) as avg_salary FROM employees GROUP BY dept", conn)`
- Compare: same result as `conn.execute(...).fetchall()` + manual DataFrame construction, but cleaner
- Runnable example: create in-memory DB, populate with employees, query into DataFrame, use pandas to sort and display

Part 2: pd.read_sql_query() and Parameters (30 min)
- Show pd.read_sql_query() (explicit query version): `pd.read_sql_query("SELECT ...", conn)`
- Show pd.read_sql_table() (reads an entire table): `pd.read_sql_table("employees", conn)` — requires SQLAlchemy connection
- Show safe parameterization with sqlite3: `pd.read_sql("SELECT * FROM emp WHERE dept = ?", conn, params=["Engineering"])`
- Show with SQLAlchemy text(): `pd.read_sql(text("SELECT * FROM emp WHERE dept = :dept"), conn, params={"dept": "Engineering"})`
- CRITICAL: show why NOT to use f-strings for SQL parameters (SQL injection example with safe alternative)
- Runnable example: parameterized query to read employees by department, called in a loop for multiple departments

Part 3: Working with the Resulting DataFrame (30 min)
- Show post-query transformations: the DataFrame from read_sql is just a normal DataFrame
- Show parsing date columns: `parse_dates=["hire_date"]` parameter in read_sql
- Show index_col parameter: `pd.read_sql("...", conn, index_col="id")`
- Show reading large tables in chunks: `for chunk in pd.read_sql("SELECT * FROM big_table", conn, chunksize=1000):`
- Runnable example: read a query result, parse dates, filter to recent hires, compute tenure in days

Part 4: Hands-on Practice (30 min)
- Exercise 1: Create a sales database (products, orders, customers). Write 3 pd.read_sql() calls to: (1) all orders with product name and customer name (JOIN query), (2) monthly revenue (GROUP BY), (3) top 5 customers by total spend. Apply pandas operations to the resulting DataFrames.
- Exercise 2: Write a parameterized query function `get_orders_by_region(conn, region: str) -> pd.DataFrame` that safely inserts the region parameter and returns orders for that region with product details.

Key Takeaways: pd.read_sql(sql, conn) → DataFrame, use params= for safe parameterization (never f-strings), parse_dates=["col"], chunksize for large queries

**lesson-07-pandas-to-sql.md** — "Writing DataFrames to SQL Tables"
Course header: SQL & Databases | 2 hours | Intermediate

Learning Objectives:
- Use df.to_sql() to write a DataFrame to a database table
- Control schema creation with if_exists and dtype parameters
- Write data in chunks for large DataFrames
- Verify writes with a read-back query

Part 1: df.to_sql() Basics (30 min)
- Show: `df.to_sql("table_name", conn, if_exists="replace", index=False)`
- Explain if_exists options: "fail" (raise if table exists), "replace" (drop + recreate), "append" (add rows)
- Explain index=False: usually don't want the pandas RangeIndex as a database column
- Show reading back to verify: `pd.read_sql("SELECT COUNT(*) as n FROM table_name", conn)`
- Runnable example: create a DataFrame of 10 employees, write to SQLite, read back and compare

Part 2: Schema Control (30 min)
- Show that to_sql() infers column types from dtypes: int64 → INTEGER, float64 → REAL, object → TEXT
- Show dtype parameter for explicit types:
  ```python
  from sqlalchemy import Integer, Text, Float, Date
  df.to_sql("table", engine, dtype={"id": Integer(), "name": Text(), "salary": Float()})
  ```
- Show how to pre-create the table with constraints (CREATE TABLE IF NOT EXISTS) and then use if_exists="append"
- Show: when to use if_exists="append" safely (only when schema matches, dedup after if needed)
- Runnable example: write a DataFrame with explicit dtype mapping, verify column types in sqlite_master

Part 3: Writing Large DataFrames (30 min)
- Show chunksize parameter: `df.to_sql("table", conn, if_exists="append", index=False, chunksize=1000)`
- Explain: for DataFrames > 10k rows, chunksize avoids memory issues with a single INSERT statement
- Show method parameter: method="multi" generates multi-row INSERT statements (faster for SQLite)
- Show wrapping to_sql in a transaction for atomicity: conn as context manager
- Runnable example: write a 1000-row DataFrame in 3 chunks of 333 rows, verify total count

Part 4: Hands-on Practice (30 min)
- Exercise 1: Build a pipeline that extracts data from a CSV string, transforms it (clean + enrich), and loads it to SQLite using to_sql. Read back and verify row count and a sample row.
- Exercise 2: Simulate an incremental load: write batch 1 to a table with if_exists="replace". Then write batch 2 with if_exists="append". Then deduplicate by id using SQL DELETE or pandas. Verify no duplicate ids remain.

Key Takeaways: to_sql(name, conn, if_exists, index=False), if_exists="replace" for full load / "append" for incremental, chunksize for large data, always verify with read-back query

Part 1: Why SQLAlchemy? (30 min)
- Explain: sqlite3 is SQLite-specific; SQLAlchemy is a unified interface for SQLite, PostgreSQL, MySQL, etc.
- Explain: changing from SQLite to PostgreSQL only requires changing the connection string
- Show connection strings: `"sqlite:///:memory:"`, `"postgresql://user:pass@host/dbname"`, `"mysql+pymysql://user:pass@host/dbname"`
- Show create_engine():
  ```python
  from sqlalchemy import create_engine
  engine = create_engine("sqlite:///:memory:", echo=False)
  ```
- Explain echo=True for debug logging (shows generated SQL) — useful in development
- Runnable example: create engine, verify connection with `engine.connect()` context manager

Part 2: Executing Raw SQL with SQLAlchemy (30 min)
- Show text() for parameterized queries:
  ```python
  from sqlalchemy import text
  with engine.connect() as conn:
      result = conn.execute(text("SELECT * FROM employees WHERE dept = :dept"), {"dept": "Engineering"})
      rows = result.fetchall()
  ```
- Show conn.execute() with text() for INSERT/UPDATE/DELETE
- Show transaction with conn.begin():
  ```python
  with engine.begin() as conn:  # auto-commits on exit, rolls back on exception
      conn.execute(text("INSERT INTO employees ..."), {...})
  ```
- Explain engine.begin() vs engine.connect(): begin() auto-commits, connect() requires explicit commit
- Runnable example: create table, insert rows with text(), query back — all via SQLAlchemy

Part 3: SQLAlchemy + pandas Integration (30 min)
- Show: pd.read_sql() accepts SQLAlchemy engine or connection (not just sqlite3 connection)
  ```python
  df = pd.read_sql("SELECT * FROM employees", engine)
  df = pd.read_sql(text("SELECT * FROM employees WHERE dept = :dept"), engine.connect(), params={"dept": "Engineering"})
  ```
- Show: df.to_sql() with SQLAlchemy engine (required for dtype parameter to work properly)
  ```python
  df.to_sql("employees", engine, if_exists="replace", index=False)
  ```
- Show: advantage of engine over raw connection — engine manages a connection pool
- Show: read_sql_table() requires SQLAlchemy engine: `pd.read_sql_table("employees", engine)`
- Runnable example: full round-trip using SQLAlchemy engine — to_sql() then read_sql() with a JOIN query

Key Takeaways: create_engine("dialect://..."), engine.begin() for auto-commit transactions, text() for parameterized SQL, use SQLAlchemy engine (not sqlite3 conn) with pandas for full feature support

```python
import pandas as pd
import sqlite3

engine = create_engine("sqlite:///:memory:", echo=False)

customers = pd.DataFrame({
    "customer_id": [1, 2, 3, 4, 5],

    "segment": ["Premium", "Standard", "Premium", "Standard", "Premium"],

})

    "product_id": [101, 102, 103, 104, 105],
    "name": ["Widget A", "Gadget Pro", "Tool Kit", "Smart Device", "Basic Pack"],
    "category": ["Hardware", "Electronics", "Hardware", "Electronics", "Accessories"],
    "unit_price": [29.99, 149.99, 49.99, 299.99, 9.99],

})

# Load to DB
customers.to_sql("customers", engine, if_exists="replace", index=False)

print("Schema loaded. Tables:", ["customers", "products", "orders"])
```

  ```python
      FROM orders o
      JOIN customers c ON o.customer_id = c.customer_id
      JOIN products p ON o.product_id = p.product_id
      GROUP BY c.segment
      ORDER BY total_revenue DESC
  """, engine)
  ```
