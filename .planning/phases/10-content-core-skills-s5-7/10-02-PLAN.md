**lesson-01-etl-overview.md** — "What Is an ETL Pipeline?"
Course header: ETL Pipelines | 2 hours | Intermediate

Learning Objectives:
- Understand Extract, Transform, Load as distinct stages
- Identify the responsibilities of each ETL stage
- Recognize common ETL pipeline shapes (batch, micro-batch, streaming overview)
- Understand why ETL code is structured as functions, not scripts

Part 1: The ETL Mental Model (30 min)
- Explain Extract: read data from source (files, APIs, databases) without transformation
- Explain Transform: apply business logic — clean, aggregate, reshape, enrich
- Explain Load: write output to destination — file, database, API
- Explain why separating concerns matters: each stage can be tested, retried, replaced independently
- Runnable example: show a naive script (all in one) vs an ETL-structured version (three functions)

```python
import pandas as pd
import io

# Naive approach: everything mixed together
def process_sales_naive(raw_data):
    df = pd.read_csv(io.StringIO(raw_data))
    df = df[df["amount"] > 0]  # filtering mixed with reading
    df["revenue"] = df["amount"] * df["price"]
    print(df.to_csv())  # output mixed with logic

# ETL approach: clear separation
def extract(raw_data: str) -> pd.DataFrame:
    """Read raw data, no business logic."""
    return pd.read_csv(io.StringIO(raw_data))

def transform(df: pd.DataFrame) -> pd.DataFrame:
    """Apply business logic."""
    df = df[df["amount"] > 0].copy()
    df["revenue"] = df["amount"] * df["price"]
    return df

def load(df: pd.DataFrame) -> str:
    """Write output."""
    return df.to_csv(index=False)

# Pipeline: compose the three stages
def run_pipeline(raw_data: str) -> str:
    return load(transform(extract(raw_data)))
```

Part 2: Pipeline Shapes (30 min)
- Batch: run once per day/hour, process entire dataset
- Incremental: process only new/changed records since last run (using watermark)
- Micro-batch: small frequent batches (Spark Structured Streaming style — conceptual only)
- Show incremental pattern: tracking a "last_processed_at" watermark
- Runnable example: simulated incremental extract — filter records by a cutoff date

Part 3: Pipeline Anatomy (30 min)
- Show a production-style pipeline with: config dict, extract(), transform(), load(), main()
- Explain entry point pattern: `if __name__ == "__main__": main()`
- Show return types: extract returns raw DataFrame, transform returns clean DataFrame, load returns row count
- Show pipeline state: start time, end time, records_in, records_out, records_failed
- Runnable example: full pipeline skeleton with timing and record counts printed

Part 4: Hands-on Practice (30 min)
- Exercise 1: Convert a provided "all-in-one" script into separate extract/transform/load functions. Verify the output is identical.
- Exercise 2: Add a pipeline_summary() function that accepts a start_time and DataFrames at each stage and prints: source rows, valid rows, rejected rows, load rows, duration seconds.

Key Takeaways: ETL separates reading/logic/writing for testability, each stage returns a value (don't print inside), pipeline function composes E→T→L

Course header: ETL Pipelines | 2 hours | Intermediate

Learning Objectives:
- Structure transformation logic as composable pure functions
- Chain multiple transformation steps using function composition
- Validate data shape and dtypes at each transformation step
- Use a transformation registry pattern for configurable pipelines

Part 1: Single-Responsibility Transforms (30 min)
- Explain: each transform function does one thing — clean, enrich, aggregate, reshape
- Show pattern: `def remove_nulls(df: pd.DataFrame) -> pd.DataFrame: return df.dropna()`
- Show chain: `df = remove_nulls(df)` → `df = add_revenue(df)` → `df = aggregate_by_region(df)`
- Discuss: functions must not have side effects, must return new DataFrame (not mutate in place)
- Runnable example: 3 chained transform functions on sales data

Part 2: Validation Inside Transforms (30 min)
- Show schema check before transform: `assert set(required_cols).issubset(df.columns)`
- Show row count guard: `assert len(df) > 0, "Empty DataFrame after filter"`
- Show dtype enforcement: cast to expected types, raise ValueError if impossible
- Show "transform result" pattern: return dict with {"data": df, "rejected": rejected_df}
- Runnable example: filter function that separates valid rows from invalid rows (negative amounts)

Part 3: Transform Composition Patterns (30 min)
- Show list-of-functions pipeline: `for step in [clean, enrich, aggregate]: df = step(df)`
- Show functools.reduce: `reduce(lambda df, fn: fn(df), transforms, initial_df)`
- Show step registry: `TRANSFORMS = {"clean": clean_fn, "enrich": enrich_fn}` configurable from YAML-like dict
- Runnable example: configurable 3-step pipeline where steps are selected from a registry by name

Part 4: Hands-on Practice (30 min)

- Exercise 2: Write a validate_schema(df, schema_dict) function where schema_dict = {"col": "dtype"}, that raises SchemaError with a clear message listing all violations.

Key Takeaways: one-responsibility transform functions, validate at boundaries, chain as list-of-functions, transform returns new DataFrame not mutated

- Show dead-letter pattern: rows that fail go to a "rejected" DataFrame with error_reason column

Key Takeaways: specific except clauses (not bare except), custom exception classes for pipeline stages, dead-letter list for partial failures, raise X from Y for chaining

**lesson-06-logging.md** — "Logging: Pipeline Audit Trails"
Course header: ETL Pipelines | 2 hours | Intermediate

Learning Objectives:

- Use appropriate log levels (DEBUG, INFO, WARNING, ERROR, CRITICAL)

- Show logging module setup:
  ```python
  import logging

      level=logging.INFO,
      format="%(asctime)s %(levelname)s %(name)s: %(message)s",
      datefmt="%Y-%m-%d %H:%M:%S"

  ```
- Explain log levels: DEBUG (detailed trace), INFO (normal progress), WARNING (recoverable issue), ERROR (stage failed), CRITICAL (pipeline must abort)
- Show: `logger.info("Extract complete: %d rows", len(df))`
- Show: `logger.warning("Null values found: %d rows removed", null_count)`
- Runnable example: simple pipeline with INFO logging at each stage boundary

Part 2: Logger Hierarchy and Module Loggers (30 min)
- Explain `getLogger(__name__)` creates a module-specific logger
- Show how to set different levels for different loggers
- Show propagation and root logger
- Show adding a FileHandler for persistent log file alongside StreamHandler
- Runnable example: two module loggers (extract_logger, transform_logger) with their own names in the log output

Part 3: Structured Pipeline Logging (30 min)
- Show a PipelineLogger context manager that logs start/end and duration:
  ```python
  class PipelineStage:
      def __init__(self, name, logger):
          self.name = name
          self.logger = logger
      def __enter__(self):
          self.start = time.time()
          self.logger.info("Stage '%s' starting", self.name)
          return self
      def __exit__(self, exc_type, exc_val, exc_tb):
          duration = time.time() - self.start
          if exc_type:
              self.logger.error("Stage '%s' failed after %.2fs: %s", self.name, duration, exc_val)
          else:
              self.logger.info("Stage '%s' complete in %.2fs", self.name, duration)
          return False  # don't suppress exceptions
  ```
- Show logging inside transforms: log record counts in, counts out, counts rejected
- Show a final pipeline summary log: total duration, total rows, stages completed
- Runnable example: pipeline with PipelineStage context manager around each ETL stage

Part 4: Hands-on Practice (30 min)
- Exercise 1: Add logging to an existing extract/transform/load pipeline. Every stage must log: stage name, rows in, rows out, duration. Errors must be logged at ERROR level with the exception message.
- Exercise 2: Build a run_summary() function that accumulates metrics from all stages and logs a final summary: "Pipeline complete: 3 stages, 1450 rows extracted, 1392 rows loaded, 58 rows rejected (4.0%), total time 0.23s".

Key Takeaways: getLogger(__name__), log at stage boundaries (INFO), log data quality issues (WARNING), log failures (ERROR), use %s formatting not f-strings in log calls (performance)

**lesson-08-idempotency-reruns.md** — "Idempotency: Safe to Re-Run"
Course header: ETL Pipelines | 2 hours | Intermediate

Learning Objectives:
- Understand idempotency: running a pipeline twice produces the same result as running it once
- Implement watermark-based incremental loading
- Use upsert patterns to prevent duplicate records
- Design pipelines that can safely recover from mid-run failure

Part 1: What Is Idempotency? (30 min)
- Explain: a pipeline is idempotent if re-running it on the same data produces no change and no duplicates
- Show why it matters: pipelines fail mid-run, cron jobs double-trigger, re-processing is common
- Show non-idempotent pattern: append-only load — doubles data on re-run
- Show idempotent pattern: overwrite-then-reload, or upsert by primary key
- Runnable example: simulate two runs of an append pipeline vs two runs of an overwrite pipeline

Part 2: Watermark-Based Incremental Processing (30 min)
