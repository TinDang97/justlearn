3. Extend the migrate function to handle version 1:
```typescript
migrate: (persistedState: unknown, version: number) => {
  if (version === 0) {
    // v0→v2: merge 12 old course dir keys into python key (same as before)
    const old = (persistedState as { completedLessons?: Record<string, string[]> })
      ?.completedLessons ?? {}
    const merged = OLD_COURSE_SLUGS.flatMap((slug) => old[slug] ?? [])
    const unique = [...new Set(merged)]
    return {
      completedLessons: unique.length > 0 ? { python: unique } : {},
    }
  }
  if (version === 1) {
    // v1→v2: data structure is already { courseSlug: string[] }, no transformation needed
    // Just carry the state forward. Storage key rename is handled by Zustand persist
    // (old key is abandoned; new key starts fresh on first write — user's v1 data is lost
    // unless we manually migrate it. However, since storage key changes, Zustand will not
    // find the old key. To preserve python progress: read from old key and seed v2.
    // Simplest approach: treat v1 state as valid v2 state since shape is identical.
    return persistedState as ProgressState
  }
  return persistedState as ProgressState
},
```

NOTE: The storage key rename ('python-course-progress' → 'justlearn-progress') means existing
localStorage data under the old key will NOT be found by Zustand on first load. The v1→v2
migration path above applies when Zustand persists with the new key and finds a v1-versioned
state. In practice, existing users will lose their progress on this key rename — that is
acceptable for a development platform at this stage (no real users yet, per STATE.md).
Document this in a comment in the file.
