<objective>
Extend the Pyodide web worker to install pandas via micropip on first need, propagating a distinct `installing` status back to the UI so users see "Installing pandas..." instead of a frozen loader.

Purpose: pandas is not bundled with Pyodide — it must be installed at runtime via micropip. Without this work, any `import pandas as pd` in a PracticeBlock raises `ModuleNotFoundError`. The install takes 3-8 seconds over CDN; the loading indicator must communicate this wait.

Output:
- Extended `pyodide.worker.mjs` with micropip install logic and install-status signalling
- Extended `RunStatus` type (`'idle' | 'loading' | 'installing' | 'ready' | 'running'`)
- Updated test coverage for the new protocol
</objective>

   `/import\s+pandas|from\s+pandas\s+import/` (covers both `import pandas as pd` and `from pandas import DataFrame`).

3. In the message handler, after Pyodide is loaded, check `needsPandas(code)`:
   - If true AND `pandasReady === null`:
     - Immediately post a status message: `self.postMessage({ id, status: 'installing' })`
     - Set `pandasReady = pyodide.loadPackage('micropip').then(() => pyodide.runPythonAsync("import micropip; await micropip.install('pandas')"))`

     - Await the existing `pandasReady` promise (handles concurrent calls gracefully)
   - If false: skip micropip entirely

4. After any pandas install step, call `runPythonAsync(code)` as before and post the final
   `{ id, output, error: null }` message.

5. Error handling: wrap the entire handler body (including micropip steps) in try/catch. On error, post
   `{ id, output, error: err.message }` — the existing error shape, no change needed in the hook.

NOTE: Pyodide 0.29.3 (used by this project — see CDN import at top of file) bundles micropip.
Use `await pyodide.loadPackage('micropip')` (not `micropip.install`) to load micropip itself, then
`await pyodide.runPythonAsync("import micropip\nawait micropip.install('pandas')")` to install pandas.
Do NOT use `micropip.install` from JS — call it from within Python via runPythonAsync to avoid async
interop issues with Pyodide's Python event loop.

1. Extend the exported `RunStatus` type:
   ```typescript
   export type RunStatus = 'idle' | 'loading' | 'installing' | 'ready' | 'running'
   ```

2. Update `sharedWorker.onmessage` handler to distinguish two message shapes:
   ```typescript
   sharedWorker.onmessage = (event: MessageEvent) => {
     const data = event.data
     // Status-only message (installing progress)
     if ('status' in data && data.status === 'installing') {
       if (isActiveRef.current) setStatus('installing')
       return  // do NOT resolve the pending promise
     }
     // Final result message
     const { id, output, error } = data
     const pending = pendingMessages.get(id)
     if (pending) {
       pendingMessages.delete(id)
       pending.resolve({ output, error })
     }
   }
   ```
   Note: `isActiveRef` is per-hook-instance but `sharedWorker.onmessage` is set globally each time a
   new worker is created. The `setStatus` call inside onmessage will always refer to the closure of the
   most recent `run()` call that created the worker. This is the existing pattern — preserve it.

3. Update `__tests__/hooks/use-pyodide-worker.test.ts`:
   - Add test: simulate worker posting `{ id, status: 'installing' }` → hook status becomes 'installing'
   - Add test: simulate worker posting `{ id, status: 'installing' }` followed by final message → status ends at 'ready'
   - Use the existing pattern of `MockWorker` and manually trigger `sharedWorker.onmessage` in tests

4. Update loading indicator in `components/code-runner/code-runner-client.tsx` — change the loading
