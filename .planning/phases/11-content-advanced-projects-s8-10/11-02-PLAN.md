**lesson-01-why-performance-matters.md**
- Level: Intermediate
- Learning Objectives: explain why pandas operations have overhead (Python GIL, object boxing), estimate processing time using row × column × operation cost model, identify the 3 most common performance anti-patterns in pandas, describe when to optimize (after correctness, at measurable bottleneck)
- Part 1 (The cost model): 1M rows × 10 columns × 1 loop iteration ≈ 10M Python object touches. Each Python object allocation ≈ 50ns → 10M × 50ns = 0.5 seconds for trivial work. Vectorized equivalent: 10M floats in contiguous C memory → one SIMD instruction per 4-8 floats → ~1000x faster. Show the mental model as a table: operation type → relative speed → example.
- Part 2 (Common anti-patterns): (1) `iterrows()` in a loop to compute a new column, (2) `apply(lambda row: ...)` for row-level arithmetic, (3) building a result list with `append()` in a loop then converting to DataFrame. For each: show the code pattern to recognize and one sentence on why it's slow.
- Part 3 (When NOT to optimize): Premature optimization quote (Knuth). Rule: profile first, optimize second. A 2-second script that runs once a day doesn't need optimization. A 2-second script that runs 10,000 times/day does. Decision tree: Does it run frequently? → Is it measurably slow? → Is it in a hot path? → Then optimize.
- Part 4 (Practice): Discussion — given 3 code snippets (loop-based, apply-based, vectorized), rank them by expected speed without running them. Explain your reasoning.
- Key Takeaways: Python loops over DataFrames are slow because they bypass C-optimized array operations, vectorization = one operation on the whole array, profile before optimizing (Knuth's rule), most pipelines have 1-2 true bottlenecks
- Common Mistakes: optimizing readable code that runs fast enough, ignoring memory (RAM limits matter more than CPU at scale), treating pandas as a database (SQL is better for joins on 100M+ rows)

**lesson-02-vectorization-with-numpy.md**
- Level: Intermediate
- Prerequisites: lesson-01, Section 2 (pandas), Data Analysis Section (NumPy basics from Python course)
- Learning Objectives: create NumPy arrays and measure their creation time, perform element-wise arithmetic without loops, use broadcasting to operate on arrays of different shapes, apply universal functions (ufuncs), explain why NumPy is faster than Python loops (contiguous memory, SIMD, no GIL per element)
- Part 1 (Before/After — loop vs array): Show timing comparison of summing a 1M-element list with Python loop vs np.sum(). Include the output showing ~100-200x speedup. Explain contiguous memory layout vs Python list of boxed objects.
- Part 2 (Vectorized arithmetic and ufuncs): `np.sqrt(a)`, `np.log(a)`, `np.exp(a)`, `np.abs(a)`. Element-wise: `a + b`, `a * b`, `a ** 2`. Show that all of these avoid loops. Emphasize: ufuncs operate on the entire array in a single C-level call.
- Part 3 (Broadcasting): `a = np.array([[1,2,3],[4,5,6]])` (shape 2×3), `b = np.array([10,20,30])` (shape 3). `a + b` → broadcasts b across rows. Rule: dimensions are compatible if equal or one of them is 1. Show 3 broadcasting examples, one that fails (incompatible shapes).
- Part 4 (Practice): PracticeBlock — given a Python list of 10,000 temperatures in Celsius, use NumPy to convert all to Fahrenheit (`F = C * 9/5 + 32`) in a single vectorized operation and print the min, max, and mean of the result
- Key Takeaways: NumPy stores data as typed C arrays (not Python objects), ufuncs call optimized BLAS/LAPACK routines, broadcasting eliminates explicit loops over dimensions, shape checking prevents silent bugs
- Common Mistakes: forgetting that NumPy arrays are typed (int64 overflow is silent), using Python loops to "help" NumPy (you're actually slowing it down), not checking shapes before broadcasting (dimension mismatch errors)

**lesson-03-pandas-vectorization-patterns.md**
- Level: Intermediate
- Prerequisites: lesson-02, Section 2 (pandas)
- Learning Objectives: replace `apply(lambda row:)` with column arithmetic, use `.str` accessor for vectorized string operations, use `.dt` accessor for vectorized datetime operations, apply conditional logic with `np.where` and `pd.cut`, measure the speedup of each pattern
- Part 1 (Column arithmetic — the most common case): Before: `df['total'] = df.apply(lambda row: row['price'] * row['quantity'], axis=1)`. After: `df['total'] = df['price'] * df['quantity']`. Timing comparison shows 50-100x speedup on 100K rows. Rule: any row-level arithmetic on numeric columns can be vectorized this way.
- Part 2 (.str accessor): Before: `df['first_name'] = df['full_name'].apply(lambda x: x.split()[0])`. After: `df['first_name'] = df['full_name'].str.split().str[0]`. Show full .str API surface: `.str.upper()`, `.str.contains()`, `.str.replace()`, `.str.len()`, `.str.strip()`. Note: .str is already vectorized — no apply needed.
- Part 3 (.dt accessor and np.where): `df['year'] = df['date'].dt.year`. `df['is_weekend'] = df['date'].dt.dayofweek >= 5`. `np.where(df['amount'] > 1000, 'high', 'low')` — vectorized conditional. `pd.cut(df['amount'], bins=[0,100,500,1000,float('inf')], labels=['low','medium','high','premium'])` — bin continuous values without loops.
- Part 4 (Practice): PracticeBlock — given a DataFrame with `full_name` (string), `birth_date` (datetime), and `salary` (float) columns: extract last name, compute age in years from birth_date using .dt, add a `salary_band` column using pd.cut — all without using apply()
- Key Takeaways: the only time apply() is acceptable is for complex multi-column logic that genuinely can't be vectorized, .str and .dt are already vectorized, np.where replaces if/else apply, pd.cut replaces range-bucketing apply
- Common Mistakes: using apply() out of habit when column arithmetic works, chaining .str calls without checking for NaN (NaN propagates through .str — use `na=False` where needed), using pd.cut without defining bin edges explicitly (auto-edges change if data changes)

**lesson-04-memory-optimization-dtypes.md**
- Level: Intermediate
- Prerequisites: lesson-01, Section 2 (dtypes)
- Learning Objectives: read a DataFrame's memory footprint with memory_usage(deep=True), downcast integer columns from int64 to the smallest fitting int type, downcast float columns, convert low-cardinality string columns to categorical dtype, measure memory reduction after downcasting
- Part 1 (Measuring memory): `df.memory_usage(deep=True)` returns bytes per column. `df.memory_usage(deep=True).sum() / 1024**2` = megabytes. Show a 1M-row DataFrame before optimization: all int64 + float64 + object → ~80MB. After optimization: int8/int16 where possible + category → ~12MB. Show the comparison table.
- Part 2 (Integer downcasting): `pd.to_numeric(df['age'], downcast='integer')` auto-selects smallest int type. Manual: if max value < 127 → use `int8`. If < 32767 → `int16`. If < 2B → `int32`. Show `df.dtypes` before and after. Unsigned integers: `uint8` for 0-255 (e.g., age, percentage).
- Part 3 (Float downcasting and categorical): `pd.to_numeric(df['price'], downcast='float')` → float32 from float64 (half the memory). Warning: float32 has ~7 decimal digits of precision — adequate for currency, not for scientific measurements. Categorical: `df['status'].astype('category')` — stores unique values once, indexes by integer internally. When to use: cardinality < 50% of rows.
- Part 4 (Practice): PracticeBlock — given a DataFrame with int64, float64, and object columns, write a `optimize_dtypes(df)` function that: (1) downcasts integers, (2) downcasts floats, (3) converts columns with < 20 unique values to category. Print memory before and after. Verify reduction is > 40%.
- Key Takeaways: default dtypes (int64, float64) are safe but wasteful, categorical dtype is the highest-impact optimization for string columns, memory reduction also speeds up operations (fewer cache misses), always validate after downcast (no data loss, no precision loss)
- Common Mistakes: downcasting to int8 without checking max value (overflow corrupts data silently), converting high-cardinality columns to category (uses MORE memory), forgetting deep=True in memory_usage (object columns show wrong size)

**lesson-05-chunked-processing.md**
- Level: Intermediate
- Prerequisites: lesson-04, Section 3 (file formats), Section 6 (ETL)
- Learning Objectives: read a large CSV in chunks using chunksize parameter, apply per-chunk transformations and accumulate results, aggregate across chunks without loading full data into memory, handle errors at the chunk level, understand the tradeoffs of chunk size selection
- Part 1 (The problem: files larger than RAM): Story — a 10GB CSV file. 16GB RAM machine. Loading fully: `pd.read_csv('big.csv')` → MemoryError. Solution: process in 100K-row chunks. Show the iterator approach: `for chunk in pd.read_csv('big.csv', chunksize=100_000):`.
- Part 2 (Accumulator pattern): Aggregating totals across chunks: initialize `total_revenue = 0.0`, `total_rows = 0`. In loop: `total_revenue += chunk['amount'].sum()`. `total_rows += len(chunk)`. After loop: `avg = total_revenue / total_rows`. Show a more complex accumulator using a defaultdict for group-level aggregation across chunks: sum by region, then compute mean after all chunks processed.
- Part 3 (Per-chunk transformation with output): When writing a cleaned version of a large file — `with open('output.csv', 'w') as f:` and inside the chunk loop: `cleaned = transform(chunk)`. First iteration: `cleaned.to_csv(f, index=False)`. Subsequent: `cleaned.to_csv(f, index=False, header=False)`. Show the `enumerate(chunks)` trick to detect first iteration.
- Part 4 (Practice): PracticeBlock — given a `generate_large_df()` function that returns a 500K-row DataFrame (simulating a large file), process it in chunks of 50K rows and compute: total row count, sum of 'amount' column, count of rows where status == 'delivered'. Compare result with full DataFrame computation to verify correctness.
- Key Takeaways: chunksize controls the memory vs. speed tradeoff (larger = faster but more RAM), chunk processing requires rethinking aggregations (two-pass: per-chunk then combine), stateless transformations (map/filter/clean) work naturally in chunks, stateful operations (median, percentile) need special handling
- Common Mistakes: choosing too-small chunks (overhead dominates), trying to compute exact median across chunks (requires loading all quantile values — use approximation), not handling the first-chunk header case when writing output CSVs

**lesson-06-profiling-and-benchmarking.md**
- Level: Intermediate
- Prerequisites: lesson-01
- Learning Objectives: time individual operations with time.perf_counter(), wrap code in a timing decorator, use cProfile to identify the slowest functions in a pipeline, interpret cProfile output (ncalls, tottime, cumtime), build a benchmark harness that runs N iterations and reports mean/min/max
- Part 1 (time.perf_counter timing): `start = time.perf_counter()`. `# ... operation ...`. `elapsed = time.perf_counter() - start`. `print(f"{elapsed:.4f}s")`. Explain why perf_counter > time.time() (higher resolution, monotonic). Build a `@timer` decorator that prints function name + elapsed time.
- Part 2 (Benchmark harness): `def benchmark(fn, *args, n=10, **kwargs):`. Run `fn(*args, **kwargs)` n times. Collect elapsed times. Return `{'mean': ..., 'min': ..., 'max': ..., 'std': ...}`. Show usage: `benchmark(process_chunk, df, n=5)`. Include a warm-up run to avoid JIT effects.
- Part 3 (cProfile basics): `import cProfile`. `cProfile.run('process_pipeline(df)')`. Reading the output: `ncalls` = how many times called, `tottime` = time in function (not in sub-calls), `cumtime` = total time including sub-calls. Rule: focus on high-cumtime functions first. Show how to sort by cumtime: `cProfile.run('...', sort='cumtime')`.
- Part 4 (Practice): PracticeBlock — given a `slow_pipeline(df)` function with 3 internal steps (one of which is artificially slow using a Python loop), use the `@timer` decorator on each step and `benchmark()` to identify the bottleneck. Report which step accounts for >80% of total time.
- Key Takeaways: measure before optimizing (you will be surprised which step is slow), perf_counter is the right tool for wall-clock timing in data pipelines, cProfile shows call graph (which function calls which slow function), optimization target: reduce cumtime of the highest-cumtime call
- Common Mistakes: timing includes I/O (disk read distorts CPU measurements — time separately), single-run timing is noisy (always average N runs), profiling the wrong granularity (too coarse misses the bottleneck, too fine adds overhead)

**lesson-07-numpy-integration-patterns.md**
- Level: Advanced
- Prerequisites: lesson-02, lesson-03, Section 2 (pandas)
- Learning Objectives: use np.where() as a vectorized if-else replacement, use np.select() for multi-condition assignment, apply np.vectorize() for functions that can't be expressed as array operations, use .values to pass pandas Series as NumPy arrays, compare performance of np.where vs apply for conditional column creation
- Part 1 (np.where — binary conditional): `np.where(condition, value_if_true, value_if_false)`. Example: `df['tier'] = np.where(df['amount'] > 1000, 'premium', 'standard')`. Before/After with apply: show 30-50x speedup. Nested np.where for 3 tiers: `np.where(a > 1000, 'premium', np.where(a > 100, 'standard', 'basic'))`.
- Part 2 (np.select — multi-condition): `conditions = [df['amount'] > 1000, df['amount'] > 100, df['amount'] > 10]`. `choices = ['premium', 'standard', 'basic']`. `df['tier'] = np.select(conditions, choices, default='micro')`. Cleaner than nested np.where for >3 conditions. Show pd.cut as an alternative when conditions are contiguous ranges.
- Part 3 (np.vectorize for non-vectorizable functions): When you genuinely need a Python function per element (e.g., external API call, complex string parsing): `vec_fn = np.vectorize(my_python_fn)`. `df['result'] = vec_fn(df['col'].values)`. Important caveat: np.vectorize is still a Python loop under the hood — it does NOT make arbitrary functions fast. It only provides a cleaner API and handles broadcasting. Use as last resort.
- Part 4 (Practice): PracticeBlock — given a transactions DataFrame, create a `risk_score` column using np.select with 4 conditions (amount > 5000 → 1.0, amount > 1000 → 0.7, amount > 100 → 0.3, default → 0.1). Then create a `flag` column using np.where (True if risk_score > 0.5). Print the value_counts of each column.
- Key Takeaways: np.where is the go-to for binary conditionals (faster than any apply), np.select handles multi-condition cleanly, np.vectorize is for API compatibility only (not performance), passing .values to NumPy avoids pandas overhead for tight numeric loops
- Common Mistakes: using np.vectorize thinking it adds speed (it doesn't, it's still Python), forgetting the `default` parameter in np.select (raises ValueError), using .values without dtypes check (object array loses NumPy optimization)

**lesson-08-optimization-in-practice.md**
- Level: Advanced
- Prerequisites: lessons 01-07
- Learning Objectives: apply the full optimization workflow (profile → identify → optimize → verify), combine dtype optimization + vectorization + chunking in a single pipeline, verify that optimizations produce identical output to the original, document optimization decisions in pipeline code
- Part 1 (The optimization workflow): Four steps as a repeatable process: (1) Profile: run benchmark() on the full pipeline, note total time and per-step breakdown. (2) Identify: find the step consuming >50% of time. (3) Optimize: apply the correct technique (loop → vectorize, large file → chunk, wide types → downcast). (4) Verify: run `pd.testing.assert_frame_equal(original_output, optimized_output)` to confirm identical results.
- Part 2 (Before — the unoptimized pipeline): Show a complete `process_orders_slow(filepath)` function that: reads the full CSV at once, uses iterrows() to compute a total column, uses apply(lambda) to classify tier, uses object dtype for all string columns. Measure with benchmark(). Expected: ~5-10s on a 100K-row dataset.
- Part 3 (After — the optimized pipeline): Show `process_orders_fast(filepath)` that: reads in chunks (chunksize=50_000), downcasts numeric columns after loading each chunk, uses column arithmetic instead of iterrows, uses np.select instead of apply. Measure with benchmark(). Expected: ~0.1-0.3s on same dataset (~20-50x speedup). Show side-by-side timing table.
- Part 4 (Practice): PracticeBlock — a `generate_orders_df(n=50_000)` helper is provided. Students apply the 4-step optimization workflow: profile the starter `slow_pipeline(df)`, identify the bottleneck by uncommenting timer decorators, apply the fastest vectorized equivalent, verify output with assert_frame_equal, print the speedup ratio.
- Key Takeaways: always profile before optimizing (the bottleneck is rarely where you expect), verify output equality after every optimization (correctness before speed), document WHY an optimization was applied (future readers need context), 80/20 rule — one or two optimizations usually account for most of the gain
- Common Mistakes: optimizing everything instead of the bottleneck, not verifying output equality (optimizations that produce wrong results faster are worse than slow correct code), removing readable code in favor of unreadable micro-optimizations for 1% gain
