**lesson-01-why-data-quality-matters.md**
- Level: Intermediate
- Learning Objectives: define the 6 data quality dimensions (completeness, accuracy, consistency, timeliness, validity, uniqueness), explain the cost of bad data with real-world examples, identify quality issues in a sample dataset by visual inspection, describe where in a pipeline quality problems originate
- Part 1 (The Cost of Bad Data): narrative + table showing quality dimension → definition → example → impact. Example: hospital misrouting a patient due to duplicate records, Amazon order going to wrong address due to stale address field.
- Part 2 (Quality Dimensions with pandas checks): code block showing how to measure each dimension on a DataFrame — `df.isnull().mean()` for completeness, `df.duplicated().sum()` for uniqueness, `pd.to_datetime(df['date'], errors='coerce').isna().sum()` for validity, value_counts on a categorical column for consistency
- Part 3 (Quality Scorecard): build a `quality_report(df)` function that returns a dict with scores per dimension (0.0-1.0). Include comparison: "score < 0.95 → flag for review"
- Part 4 (Practice): PracticeBlock — given a messy transactions DataFrame, call `quality_report(df)` and identify which dimensions are below 0.95
- Key Takeaways: bad data costs companies $12.9M/year on average (Gartner), quality must be measured not assumed, shift-left = catch issues at ingestion not reporting
- Common Mistakes: checking quality only at end of pipeline, treating nulls as always bad (sometimes expected), ignoring referential consistency

**lesson-02-schema-validation-with-pandera.md**
- Level: Intermediate
- Prerequisites: lesson-01, pandas DataFrames (Section 2)
- Learning Objectives: install and import pandera, define a DataFrameSchema with Column and Check, run validation against a DataFrame and catch SchemaError, use built-in checks (greater_than, isin, str_matches), write custom Check functions
- Part 1 (DataFrameSchema basics): `import pandera as pa`. Define schema for an orders DataFrame: `order_id` (int, nullable=False), `amount` (float, pa.Check.greater_than(0)), `status` (str, pa.Check.isin(['pending','shipped','delivered'])). Show `schema.validate(df)` success and then inject a bad row to trigger SchemaError.
- Part 2 (Custom checks and lazy validation): `pa.Check(lambda s: s.str.len() <= 50, error="name too long")`. Lazy validation: `schema.validate(df, lazy=True)` collects ALL errors instead of failing on first. Show how to iterate `exc.failure_cases`.
- Part 3 (Schema from DataFrame inference): `pa.infer_schema(df)` — useful for bootstrapping. Show output and how to refine. Mention `schema.to_script()` to export as Python code.
- Part 4 (Practice): PracticeBlock — validate an employees DataFrame against a pre-written schema; the DataFrame has 3 injected violations; student must read the SchemaError and fix the data (not the schema)
- Key Takeaways: pandera schemas are code, version them alongside pipeline code; lazy=True is essential for batch reporting; infer_schema bootstraps but always review before production
- Common Mistakes: validating after transformation instead of before, using nullable=True everywhere (defeats purpose), not handling SchemaError in production code

**lesson-03-data-profiling-with-pandas.md**
- Level: Intermediate
- Prerequisites: lesson-01, Section 2 (DataFrame statistics)
- Learning Objectives: generate a null rate report for every column, compute value distribution for categorical columns, detect columns with near-zero variance, build a reusable profile_dataframe() function, interpret profiling output to prioritize cleaning work
- Part 1 (Null analysis): `df.isnull().sum()` and `df.isnull().mean().sort_values(ascending=False)`. Show a bar chart of null rates as text (use `*` characters to simulate a bar — no matplotlib dependency in browser). Highlight threshold: >20% nulls → candidate for dropping.
- Part 2 (Distribution profiling): `df.describe(include='all')`. For each object column: `df[col].value_counts(normalize=True).head(10)`. Detect high-cardinality: `df[col].nunique() / len(df) > 0.9` → likely ID column or free text.
- Part 3 (profile_dataframe() function): Build a function that returns a DataFrame where each row is a column profile: `{'column', 'dtype', 'null_rate', 'unique_count', 'top_value', 'top_freq'}`. Load `data/transactions.csv` and run the profile.
- Part 4 (Practice): PracticeBlock — run `profile_dataframe()` on a provided DataFrame, identify the 2 columns with the highest null rate and the 1 column that appears to be a free-text field based on cardinality
- Key Takeaways: profiling is always step 1 before cleaning, high null rate doesn't always mean bad data (sparse features), cardinality reveals column semantics
- Common Mistakes: jumping to cleaning without profiling, treating describe() as sufficient profiling, profiling a sample instead of the full dataset for schema inference

**lesson-04-detecting-anomalies-and-outliers.md**
- Level: Intermediate
- Prerequisites: lesson-03, Section 2 (pandas stats)
- Learning Objectives: apply IQR method to flag numeric outliers, use z-score to detect statistical outliers, build boolean masks to isolate anomalous rows, distinguish between outliers (statistical) and anomalies (domain-specific), decide when to remove vs. cap vs. investigate outliers
- Part 1 (IQR method): `Q1, Q3 = df['amount'].quantile([0.25, 0.75])`. `IQR = Q3 - Q1`. `lower = Q1 - 1.5*IQR`, `upper = Q3 + 1.5*IQR`. `outliers = df[(df['amount'] < lower) | (df['amount'] > upper)]`. Show count before/after.
- Part 2 (Z-score method): `from scipy import stats` is NOT available in Pyodide by default — implement manually: `z_scores = (df['amount'] - df['amount'].mean()) / df['amount'].std()`. Flag `|z| > 3`. Compare IQR vs z-score counts on same data.
- Part 3 (Domain anomalies): Beyond statistics — negative order amounts, dates in the future, age > 150. Pattern: `df[df['age'] > 100]`, `df[df['amount'] < 0]`, `df[df['order_date'] > pd.Timestamp.today()]`. Show how to build a `check_domain_rules(df)` function returning a summary.
- Part 4 (Practice): PracticeBlock — given a sales DataFrame with embedded outliers (5 rows with amount > 10,000 in a dataset where normal range is 10-500), use IQR to detect them, print the suspicious rows, and cap them at the upper fence
- Key Takeaways: IQR is robust (not sensitive to extremes), z-score assumes normality, domain rules catch what statistics miss, always investigate before dropping
- Common Mistakes: removing outliers without investigation (valid extreme values exist), using mean/std on skewed data, applying row-level outlier removal to time series (temporal context matters)

**lesson-05-data-contracts-introduction.md**
- Level: Advanced
- Prerequisites: lessons 01-02
- Learning Objectives: explain what a data contract is and why it exists, identify the three layers of a contract (schema, semantics, SLA), describe the producer/consumer model in pipeline architecture, map contracts to real pipeline boundaries (ingestion → processing → reporting), differentiate contract enforcement from schema validation
- Part 1 (What is a data contract): Analogy — a REST API has OpenAPI spec (the contract). A data pipeline should too. Contract = agreed-upon spec between the team that produces data and the team that consumes it. Show a narrative table: Schema layer (field names, types, nullability), Semantics layer (business rules: "amount is always in USD", "status values are an enum"), SLA layer (freshness: "updated within 1 hour", row count: "at least 1000 rows per run").
- Part 2 (Why contracts prevent pipeline failures): Story: upstream team renames `user_id` to `userId` → downstream dashboard breaks silently for 3 days. With a contract in place, the rename would cause an immediate contract violation at the pipeline boundary. Show timeline diagram as ASCII art.
- Part 3 (Contract layers as code — preview): Tease the pandera + dataclass pattern. Show a minimal example of what a contract looks like as a Python class (full implementation in lesson-06). No runnable code yet — conceptual.
- Part 4 (Practice): Discussion exercise — given 3 pipeline diagrams (text-based), identify where a contract boundary should be placed and what the contract should assert. No code required.
- Key Takeaways: contracts make implicit agreements explicit, they live in version control alongside pipeline code, contract violations at boundaries are better than silent data corruption downstream
- Common Mistakes: treating contracts as documentation (they must be enforced), defining contracts too broadly (one contract per boundary, not one mega-contract), enforcing contracts in prod but not in tests

**lesson-06-implementing-data-contracts.md**
- Level: Advanced
- Prerequisites: lesson-05, lesson-02 (pandera)
- Learning Objectives: implement a data contract as a Python dataclass with embedded pandera schema, write an enforce_contract() function that validates at a pipeline boundary, raise a ContractViolationError with actionable details, version a contract using a version field, test a contract in isolation with pytest
- Part 1 (Contract as dataclass): Define `@dataclass class OrdersContract:` with fields `version: str = "1.0"`, `schema: pa.DataFrameSchema = field(default_factory=lambda: pa.DataFrameSchema({...}))`, `min_row_count: int = 1`, `max_null_rate: float = 0.05`. Method `enforce(df: pd.DataFrame) -> None`.
- Part 2 (enforce() implementation): Inside `enforce()`: run `self.schema.validate(df, lazy=True)`, check `len(df) >= self.min_row_count`, check `df.isnull().mean().max() <= self.max_null_rate`. Collect all violations. If any, raise `ContractViolationError(f"Contract {self.version} violated: {violations}")`. Show full code.
- Part 3 (Wiring into a pipeline): Show where to call `contract.enforce(df)` in a typical ETL function: after extract (raw contract), after transform (processed contract), before load (load contract). Annotate each call site with a comment explaining what guarantee it provides.
- Part 4 (Practice): PracticeBlock — a pre-written `transform_orders(df)` function is given. Student must: define an `OrdersContract`, inject a row that violates `min_row_count=100` (start with 50-row DataFrame), call `contract.enforce(df)`, catch `ContractViolationError` and print the message
- Key Takeaways: contracts are value objects (immutable, version-stamped), `enforce()` at boundaries = shift-left quality, ContractViolationError should include enough detail to fix the data (not just "contract failed")
- Common Mistakes: swallowing ContractViolationError with bare except, not versioning contracts (v1 vs v2 ambiguity), enforcing contracts only at the end of the pipeline

**lesson-07-testing-pipelines-with-pytest.md**
- Level: Advanced
- Prerequisites: lesson-06, Python testing basics
- Learning Objectives: write pytest fixtures that create sample DataFrames, use pytest.mark.parametrize to test multiple input variants, use pd.testing.assert_frame_equal to compare DataFrame outputs, test that invalid inputs raise expected exceptions, structure a pipeline test file with fixture → act → assert pattern
- Part 1 (pytest fixtures for DataFrames): `@pytest.fixture def sample_orders():` returns a 5-row DataFrame with all valid data. `@pytest.fixture def dirty_orders(sample_orders):` adds 2 rows with nulls and 1 with negative amount. Show how to use fixtures in test functions.
- Part 2 (parametrize for edge cases): `@pytest.mark.parametrize("amount,expected_valid", [(10.0, True), (0.0, False), (-5.0, False), (999999.0, True)])`. Show a `test_validate_amount()` test that runs all 4 cases. Explain why parametrize > copy-paste.
- Part 3 (assert_frame_equal for transformations): Import `from pandas.testing import assert_frame_equal`. Build `test_normalize_amounts()` that creates an input DataFrame, calls `normalize_amounts(df)`, and asserts exact output with `check_dtype=True, check_exact=False, rtol=1e-5`. Show how to use `assert_series_equal` for single-column checks.
- Part 4 (Testing exception paths): `with pytest.raises(ContractViolationError, match="min_row_count"):` — test that a pipeline raises the right error for the right reason. Show `pytest.raises` as context manager. Also show `pytest.approx` for floating-point comparisons.
- Key Takeaways: fixtures are reusable setup (DRY principle), parametrize eliminates copy-paste tests, assert_frame_equal gives column-level diff on failure, test the contract not the framework
- Common Mistakes: building fixtures that are too large (50+ rows make failures hard to debug), not using check_dtype in assert_frame_equal (silent type promotions), testing implementation details instead of behavior

**lesson-08-data-quality-dashboard.md**
- Level: Advanced
- Prerequisites: lessons 01-04
- Learning Objectives: aggregate per-column quality metrics into a single quality score, format a quality report as a structured text table, define quality thresholds (green/yellow/red), build a `run_quality_check(df, thresholds)` function that returns a pass/fail summary, integrate the dashboard into a simulated pipeline run
- Part 1 (Quality score aggregation): Define dimensions and weights: `WEIGHTS = {'completeness': 0.3, 'validity': 0.3, 'uniqueness': 0.2, 'consistency': 0.2}`. For each dimension, compute a 0.0-1.0 score using pandas checks from lesson-01 through 04. Weighted average = overall score.
- Part 2 (Formatted report): Build `format_quality_report(results: dict) -> str`. Output must look like:
  ```
  ┌─────────────────────────────────────┐
  │  DATA QUALITY REPORT                │
  │  Dataset: transactions.csv          │
  │  Rows: 1,000  |  Columns: 8        │
  ├──────────────────┬────────┬─────────┤
  │ Dimension        │ Score  │ Status  │
  ├──────────────────┼────────┼─────────┤
  │ Completeness     │ 0.97   │  PASS   │
  │ Validity         │ 0.89   │  WARN   │
  │ Uniqueness       │ 1.00   │  PASS   │
  │ Consistency      │ 0.72   │  FAIL   │
  ├──────────────────┼────────┼─────────┤
  │ Overall Score    │ 0.90   │  WARN   │
  └──────────────────┴────────┴─────────┘
  ```
  Use Python string formatting (no external libraries).
- Part 3 (Thresholds and action rules): `THRESHOLDS = {'pass': 0.95, 'warn': 0.80}`. Status = "PASS" if score >= pass, "WARN" if >= warn, "FAIL" otherwise. Define `run_quality_check(df, thresholds) -> dict` that returns `{'overall': float, 'status': str, 'dimensions': dict, 'blocking': bool}` where `blocking=True` if any dimension is FAIL.
- Part 4 (Practice): PracticeBlock — load `data/transactions.csv`, run `run_quality_check(df, THRESHOLDS)`, print the formatted report, assert that the overall score is above 0.80, print "Pipeline blocked" if `blocking=True`
- Key Takeaways: quality dashboards should be runnable in CI not just ad-hoc, weighted scoring lets you prioritize dimensions by business impact, blocking=True prevents bad data from reaching downstream consumers
- Common Mistakes: hardcoding thresholds (make them configurable), computing quality on a sample (use full dataset), not logging the report to a file (ephemeral output is useless for debugging)
