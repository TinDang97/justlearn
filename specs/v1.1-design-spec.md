# JustLearn v1.1 -- UX/UI Design Specification

**Version:** 1.1.0
**Date:** 2026-03-14
**Author:** Tin Dang
**Status:** Draft

---

## 1. Design Brief

JustLearn v1.1 transforms a functional 12-course listing into a cohesive, single-course learning experience with a Medium-quality reading surface, chunked lesson content, and a branded homepage. The design targets complete Python beginners (300 students) and must feel welcoming, calm, and confidence-building -- never intimidating.

The visual direction draws from three reference points: **Medium** (reading comfort, typographic excellence), **Notion** (clean structural hierarchy, sidebar navigation), and **Brilliant/Codecademy** (learning progression, interactive warmth).

---

## 2. Target Persona

**Name:** Alex, 23
**Background:** College student or early-career professional with zero programming experience. Possibly apprehensive about code. Browses on laptop during evenings, occasionally on phone during commute.
**Goals:** Learn Python from scratch, feel a sense of progress, not get overwhelmed.
**Pain points:** Long walls of text feel exhausting. Too many courses feel scattered. Doesn't know where to start.
**Tech comfort:** Comfortable with web browsing, Google Docs, social media. Has never used a terminal.
**Emotional needs:** Encouragement, clarity, bite-sized wins.

## 5. Typography System

### Font Selection

**Primary: Inter** (already in use -- keep it)
- Rationale: Excellent screen readability at all sizes, large x-height, open apertures. Well-suited for educational content. Already loaded via `@fontsource/inter`.

**Display/Heading alternative: Source Serif 4** (Google Fonts, free)
- Rationale: Pairs beautifully with Inter. Serif headings create a Medium/editorial feel that signals "this is something worth reading." Gives lesson titles gravitas without formality.
- Trade-off: Adding a second font increases load. If bundle size is a concern, use Inter for everything and differentiate headings via weight and size only.

**Monospace: JetBrains Mono** (Google Fonts, free)
- Rationale: Purpose-built for code. Ligatures for `!=`, `=>`, `<=`. Tall x-height matches Inter's proportions. Better than browser defaults for beginner code comprehension.
- Current state: No dedicated monospace is loaded; browser defaults are used.

### Type Scale (4px grid aligned)

| Token | Size | Line Height | Weight | Usage |
|-------|------|-------------|--------|-------|
| `--text-display` | 2.5rem (40px) | 1.2 (48px) | 700 | Homepage hero title |
| `--text-h1` | 2rem (32px) | 1.25 (40px) | 700 | Lesson title |
| `--text-h2` | 1.5rem (24px) | 1.33 (32px) | 600 | Section headings within lesson |
| `--text-h3` | 1.25rem (20px) | 1.4 (28px) | 600 | Subsection headings |
| `--text-h4` | 1.125rem (18px) | 1.44 (26px) | 600 | Minor headings |
| `--text-body` | 1.125rem (18px) | 1.75 (32px) | 400 | Lesson prose -- larger than typical for reading comfort |
| `--text-body-small` | 1rem (16px) | 1.625 (26px) | 400 | UI text, sidebar items |
| `--text-caption` | 0.875rem (14px) | 1.43 (20px) | 400 | Metadata, badges, timestamps |
| `--text-overline` | 0.75rem (12px) | 1.33 (16px) | 600 | Section labels, uppercase tracking |
| `--text-code` | 0.9375rem (15px) | 1.6 (24px) | 400 | Code blocks |
| `--text-code-inline` | 0.875rem (14px) | inherit | 500 | Inline code |

### Typography CSS Custom Properties

```css
--font-sans: 'Inter', system-ui, -apple-system, sans-serif;
--font-serif: 'Source Serif 4', Georgia, 'Times New Roman', serif;
--font-mono: 'JetBrains Mono', 'Fira Code', 'Cascadia Code', monospace;

--letter-spacing-tight: -0.02em;    /* display/h1 */
--letter-spacing-normal: 0;          /* body */
--letter-spacing-wide: 0.05em;       /* overline/section labels */
```

### Key Typography Decisions

1. **Lesson body text at 18px** (not 16px). For educational long-form content, 18px dramatically improves reading comfort. Medium uses 21px; 18px is the right balance for technical content with code blocks.

2. **Line height 1.75 for body prose**. Already set in `globals.css` -- keep it. Generous leading reduces the "wall of text" feeling that discourages beginners.

3. **Optimal line length: 65ch** (already set via `max-w-[65ch]` in lesson page). This is correct and should be preserved.

4. **Heading hierarchy via weight, not just size**. Use 700 for h1, 600 for h2-h4. This creates clear visual hierarchy without oversized headings.

## 8. Course Structure UX -- Unified Course with Sections

### Information Architecture Change

**Current:** 12 separate courses, each with 10 lessons. User sees a grid of 12 cards.
**Proposed:** 1 course ("Python -- From Zero to Confident") with 12 sections, each containing 10 lessons. User sees a structured curriculum.

### URL Structure

```
/                           -- Homepage (new)
/course                     -- Course overview (replaces /courses)
/course/fundamentals        -- Section detail (optional, could be anchor-based)
/course/fundamentals/what-is-programming  -- Lesson page
```

Alternative (simpler, recommended for v1.1):
```
/                           -- Homepage
/course                     -- Course overview with all sections listed
/course/lesson-01           -- Flat lesson numbering (1-122)
```

Trade-off analysis:
- **Nested URLs** (`/course/section/lesson`): Clearer hierarchy but deeper nesting, more routing complexity.
- **Flat URLs** (`/course/lesson-01`): Simpler routing, easier deep linking, but loses section context in URL.
- **Recommendation:** Keep the current `/courses/[courseSlug]/[lessonSlug]` structure but rename it conceptually. Each "course" becomes a "section". The `/courses` page becomes the course overview. This minimizes migration effort while the UX communicates "one unified course."

### Course Overview Page (`/courses` -> reimagined as `/course`)

**Layout:** Single column, `max-w-3xl`, with clear visual hierarchy.

```
[Page]

  [Course Header]
  "Python -- From Zero to Confident"  -- h1, display size
  "A complete beginner's guide to Python programming. 12 sections. 122 lessons.
   From your first line of code to building real projects."
  [Overall progress bar] -- full width, shows X/122 lessons completed

  [Section List -- vertical, accordion-style]
  Each section is a collapsible card:

    [Section Card]
    Border-left: 3px solid (green if started, muted if not started)

      [Header row -- always visible, clickable to expand]
      [Number] "01"  [Title] "Python Fundamentals"  [Meta] "10 lessons"
      [Progress] "3/10" or checkmark if complete
      [Chevron] expand/collapse indicator

      [Expanded content -- lesson list]
      Ordered list of lessons with:
      - Completion icon (circle / check-circle)
      - Lesson number
      - Lesson title
      - Duration badge

      Links to individual lesson pages.

  [Default state]: First incomplete section expanded, rest collapsed.
  [Completed sections]: Collapsed with a green checkmark badge.
```

**Rationale:** Accordion pattern shows the full scope of the curriculum at a glance (answering "how long will this take?") while keeping the view scannable. The left-border color coding creates an immediate visual map of progress.

## 9. Lesson Chunking UX

### Problem

Current lessons are 2 hours long, rendered as a single continuous markdown document (~340 lines). This creates:
1. Scroll fatigue -- users lose their place
2. No sense of progress within a lesson
3. No natural stopping points

### Solution: Lesson Patches

Each lesson is divided into **patches** (3-5 per lesson, roughly 15-25 minutes each). Patches are semantic content divisions, not arbitrary cuts.

### Terminology

- **Section** = what was a "course" (e.g., "Python Fundamentals")
- **Lesson** = individual lesson (e.g., "What is Programming?")
- **Patch** = chunk within a lesson (e.g., "Part 1: What is a Computer Program?")

### Lesson Page Layout

```
[Lesson Page]
Max-width: 65ch (prose) with sidebar offset on desktop

  [Breadcrumb]
  Course > Python Fundamentals > What is Programming?

  [Lesson Header]
  [Badge: "Section 1"] [Badge: "Lesson 1 of 10"] [Duration: "2 hours"]
  [H1] "What is Programming?"

  [Table of Contents -- sticky sidebar on desktop, collapsible top-bar on mobile]
  Lists all patches within the lesson:

    Part 1: What is a Computer Program?  [active/current indicator]
    Part 2: What is Python?
    Part 3: High-Level vs Machine Language
    Part 4: Tools You'll Need
    Practice: Mini Exercises

  Progress through patches shown via filled/unfilled dots or a vertical progress line.

  [Lesson Content -- prose area]
  Patches are rendered as continuous content (single page scroll, not paginated).
  Each patch is separated by:
  - A horizontal rule (thematic break)
  - A patch heading (h2 level)
  - A subtle "Part X of Y" label above the heading

  [Inline practice block] -- after each patch or at natural breakpoints
  Collapsible "Try it yourself" section with:
  - A prompt/question
  - Embedded code runner (existing component)
  - Expandable hint
  - Expandable solution

  [End-of-lesson block]
  - Concept map (existing mindmap component)
  - Practice exercises (existing code runner with exercises)
  - "Mark Complete" button
  - Prev/Next lesson navigation
```

### Table of Contents Component

**Desktop (lg+):**
- Fixed position on the right side of the content area, outside the 65ch column
- Vertically aligned list of patch titles
- Active patch highlighted (detected via Intersection Observer on h2 headings)
- Uses `position: sticky; top: 5rem;` within a right-margin area
- Width: 200-240px
- Typography: `--text-caption`, `--color-foreground-muted`, active item in `--color-foreground` with `font-medium`

**Mobile (<lg):**
- Collapsible section below the lesson header, above the content
- Shows current patch as a compact bar: "Part 2 of 5: What is Python?"
- Tap to expand full TOC as a dropdown/sheet
- Compact height: 44px collapsed

### Scroll Progress Indicator

A thin (2-3px) horizontal progress bar fixed at the very top of the viewport (below the site header), colored with `--color-primary`. Tracks scroll position within the lesson. Provides a constant sense of "I'm making progress."

### Patch Divider Design

```
[Patch Divider]
Margin: 48px top, 32px bottom
Content:
  [Thin horizontal line -- 1px, --color-border, 80px wide, centered]
  [Label] "Part 2 of 5" -- overline style, centered, --color-foreground-subtle
  [H2 heading] "What is Python?" -- normal heading style
```

## 10. Reading Experience

### Prose Styling (Tailwind Typography Overrides)

The existing `@tailwindcss/typography` plugin is in use. Override the default `prose` styles:

```css
/* Lesson article styling */

  --tw-prose-body: var(--color-foreground);
  --tw-prose-headings: var(--color-foreground);
  --tw-prose-links: var(--color-accent);
  --tw-prose-bold: var(--color-foreground);
  --tw-prose-code: var(--color-foreground);

  --tw-prose-quote-borders: var(--color-primary);

  font-size: 1.125rem;           /* 18px body */
  line-height: 1.75;             /* 32px -- already set */
}

/* Headings within lessons */
.prose h2 {
  font-size: 1.5rem;
  margin-top: 3rem;              /* generous spacing before major sections */
  margin-bottom: 1rem;
  font-weight: 600;
  letter-spacing: -0.01em;
  scroll-margin-top: 5rem;       /* accounts for sticky header when jumping via TOC */
}

.prose h3 {
  font-size: 1.25rem;
  margin-top: 2rem;
  margin-bottom: 0.75rem;
  font-weight: 600;
}

/* Paragraph spacing */
.prose p {
  margin-bottom: 1.5rem;         /* 24px -- breathing room between paragraphs */
}

/* Lists */
.prose ul, .prose ol {
  margin-top: 0.75rem;
  margin-bottom: 1.5rem;
  padding-left: 1.5rem;
}

.prose li {
  margin-bottom: 0.5rem;

/* Blockquotes -- used for "key concept" callouts */

  border-radius: 0 var(--radius-md) var(--radius-md) 0;
  font-style: normal;            /* override italic default */
  color: var(--color-foreground);
}

/* Inline code */
.prose code:not(pre code) {
  background: var(--color-background-muted);
  padding: 0.125rem 0.375rem;
  border-radius: var(--radius-sm);
  font-family: var(--font-mono);
  font-size: 0.875rem;
  font-weight: 500;
  color: var(--color-foreground);
}

/* Strong emphasis for key terms */
.prose strong {
  font-weight: 600;
  color: var(--color-foreground);
}

/* Links */
.prose a {
  color: var(--color-accent);
  text-decoration: underline;
  text-underline-offset: 3px;
  text-decoration-thickness: 1px;
  transition: color 150ms;
}

.prose a:hover {
  color: var(--color-accent-hover);
}
```

### Reading Comfort Features

1. **Scroll-margin on headings**: All h2/h3 elements get `scroll-margin-top: 5rem` so TOC jumps don't hide content behind the sticky header.

2. **Smooth scroll**: Apply `scroll-behavior: smooth` to the lesson content area (not globally, to avoid interfering with instant navigation on route changes).

3. **Image handling**: Lesson images (if any) should be `rounded-lg` with optional captions styled as `--text-caption`, centered, `--color-foreground-muted`.

4. **Callout blocks**: In addition to blockquotes, consider custom MDX components for:
   - **Tip** (green left-border, lightbulb icon)
   - **Warning** (amber left-border, alert icon)
   - **Key concept** (blue left-border, bookmark icon)

### Syntax Highlighting Theme

| Token | Light Color | Dark Color |
|-------|-------------|------------|

## 13. Motion and Animation

### Duration Tokens

```css
--duration-fast: 100ms;      /* hover color changes, opacity */
--duration-normal: 200ms;    /* transforms, shadows */
--duration-slow: 300ms;      /* expanding sections, page transitions */
--duration-slower: 500ms;    /* progress bar fills, celebration animations */
```

### Easing

```css
--ease-out: cubic-bezier(0.16, 1, 0.3, 1);      /* elements appearing */
--ease-in-out: cubic-bezier(0.65, 0, 0.35, 1);   /* symmetric transforms */
--ease-spring: cubic-bezier(0.34, 1.56, 0.64, 1); /* bouncy micro-interactions */
```

### Interaction Animations

| Element | Trigger | Animation |
|---------|---------|-----------|
| Card hover | mouseenter | `translateY(-1px)`, `shadow-md`, `--duration-normal` |
| Button press | click | `scale(0.97)` spring (existing, keep it) |
| Copy button | click -> "Copied!" | Fade in checkmark, fade out after 2s |
| Progress bar | value change | Width transition, `--duration-slower`, ease-out |
| TOC active item | scroll | Highlight color transition, `--duration-fast` |
| Sidebar accordion | toggle | Height slide, `--duration-slow`, ease-out |
| Lesson scroll progress | scroll | Width update via RAF (no CSS transition, direct style) |
| Mark Complete | click | Scale bounce + confetti burst (tiny, 500ms) |

### Motion Preferences

All animations wrapped in `motion-safe:` media query (already used in pilot-space). Users with `prefers-reduced-motion: reduce` get instant state changes.

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

## 15. Accessibility Annotations

### Focus Management

- **Tab order**: Header nav -> Sidebar (desktop) -> Main content area -> Footer nav
- **Skip link**: Add a visually hidden "Skip to content" link as the first focusable element
- **Focus trapping**: Modal dialogs (search, mobile sidebar sheet) trap focus
- **Focus-visible ring**: 2px `--color-primary`, 2px offset, on all interactive elements

### ARIA Patterns

| Component | Pattern | Notes |
|-----------|---------|-------|
| Sidebar accordion | `role="tree"` with `aria-expanded` | Sections are tree items |
| TOC | `role="navigation"` with `aria-label="Table of contents"` | Current item: `aria-current="true"` |
| Progress bar | `role="progressbar"` with `aria-valuenow`, `aria-valuemin`, `aria-valuemax` | Current impl uses div -- needs ARIA |
| Lesson complete button | `aria-pressed` toggle | Already has distinct visual states |
| Code block copy | `aria-label="Copy code"` (already present) | Add `aria-live="polite"` for "Copied!" feedback |
| Theme toggle | `aria-label` with current state | "Switch to dark mode" / "Switch to light mode" |

### Screen Reader Considerations

1. **Lesson headings**: Ensure heading hierarchy is correct (h1 for lesson title, h2 for patches, h3 for subsections). Current MDX content uses `#` through `####` which maps correctly.
2. **Code blocks**: Consider `aria-label="Python code example"` on the `<pre>` element.
3. **Progress indicators**: Numeric values, not just visual bars. "3 of 10 lessons completed" as text.
4. **Decorative icons**: `aria-hidden="true"` on decorative icons (already done for lucide icons in most cases).
